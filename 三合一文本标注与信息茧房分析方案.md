# 三合一文本标注与信息茧房分析方案

## 1. 研究主题

本项目建议围绕以下主题展开：

```text
西贝事件在 B 站视频评论区中的舆论聚集、立场分化与信息茧房倾向分析
```

需要注意，本项目的数据来自 B 站相关视频评论区，因此研究对象应表述为：

```text
B 站视频评论区中的局部舆论结构
```

不宜直接表述为：

```text
全网社会舆论
完整平台推荐机制下的信息茧房
```

更准确的分析目标是：

```text
在西贝事件这一公共争议事件中，B 站评论区是否出现了议题集中、情绪共振、立场倾斜和同立场互动等现象。
```

## 2. 核心研究问题

本项目可以围绕四个研究问题展开：

```text
RQ1：评论区主要围绕哪些议题展开讨论？
RQ2：评论区的情绪表达和立场分布是否呈现集中化？
RQ3：高赞评论是否强化了某些议题、情绪或立场？
RQ4：用户回复网络是否表现出同立场互动或局部同温层倾向？
```

这四个问题共同服务于一个综合判断：

```text
该事件在 B 站评论区中是否呈现事件型信息茧房倾向。
```

## 3. 数据基础

当前可用的数据底座来自 `data/layer1/` 和 `data/database/xibei_event.db`。

核心事实表包括：

```text
sources：视频源
users：评论用户
contents：一级评论与楼中楼回复
relations：评论视频关系与用户回复关系
```

其中：

```text
contents.csv / contents 表
```

用于文本分析，包括主题、情绪、立场识别。

```text
relations.csv / relations 表
```

用于互动网络分析，包括谁回复谁、是否跨立场互动、是否形成同质化互动结构。

当前全量结构化数据规模约为：

```text
视频源：32 个
评论与回复：40,803 条
用户：28,343 个
关系记录：40,803 条
```

## 4. 为什么需要文本标注

信息茧房倾向不能只靠评论数量和点赞数判断。

要判断评论区是否存在议题聚集、情绪共振和立场同质化，需要先给每条评论补充三个文本标签：

```text
topic_label：评论在讨论什么
sentiment_label：评论以什么情绪表达
stance_label：评论站在哪一边或指向谁
```

这三个标签生成后，才能进一步计算：

```text
主题集中度
情绪一致性
立场一致性
高赞评论立场集中度
同立场回复比例
跨立场回复比例
```

互动关系本身可以直接从 `relations` 得到，但“同立场互动”必须依赖 `stance_label` 才能判断。

## 5. 三合一文本标注方法

本项目建议采用：

```text
规则初标 + DeepSeek 校正 + MacBERT 微调
```

三种方法不是互相替代，而是各自承担不同功能。

## 6. 方法一：规则初标

规则初标基于关键词和事件知识进行分类。

例如：

```text
预制菜、现做、冻货、中央厨房 -> 预制菜与食材真实性
太贵、价格、性价比、21 元馒头 -> 价格与性价比
doge、笑死、绷不住 -> 调侃/讽刺
西贝太贵、翻车、自爆 -> 质疑西贝
```

规则初标的作用是：

```text
1. 给全量评论提供初始标签
2. 保留可解释的关键词依据
3. 筛选高置信度样本
4. 找出低置信度、多候选标签和需要复核的样本
```

规则法的优点是透明、稳定、成本低。

规则法的不足是难以处理：

```text
反讽
玩梗
隐含立场
多主题混合
上下文依赖表达
```

因此，规则不适合作为最终唯一方法。

## 7. 方法二：DeepSeek 校正

DeepSeek 负责处理规则难以判断的语义样本。

优先交给 DeepSeek 的样本包括：

```text
高赞评论
规则低置信度评论
规则命中多个候选标签的评论
包含明显反讽、玩梗、表情符号的评论
少数类标签候选评论
每个视频中的代表性抽样评论
```

DeepSeek 的作用是：

```text
1. 校正规则误判
2. 处理反讽和复杂语义
3. 形成较高质量的弱标注数据集
4. 为 MacBERT 微调提供训练标签
```

DeepSeek 的不足是：

```text
1. 调用成本高于本地模型
2. 大批量处理速度较慢
3. 结果需要保存原始响应，保证可追溯
4. 关键样本仍建议人工抽检
```

因此，DeepSeek 更适合作为标注教师和疑难样本复核工具，而不是长期全量分析工具。

## 8. 方法三：MacBERT 微调

MacBERT 负责把前两步形成的标注知识固化成本地分类模型。

微调后的模型用于：

```text
1. 对 40,803 条评论进行全量预测
2. 降低后续重复分析的 API 成本
3. 提高分析流程的可复现性
4. 为新增评论数据提供快速自动标注能力
```

本项目中的 `Chinese_macbert_base/` 是中文 MacBERT-base 预训练模型资源。

微调时可以使用两种方式：

```text
方式 A：直接使用 Hugging Face 的 hfl/chinese-macbert-base
方式 B：转换并使用本地 TensorFlow checkpoint
```

实际实现上，优先建议方式 A，因为更容易接入 PyTorch 与 Transformers 训练流程。

## 9. 标签任务设计

建议拆成三个监督分类任务。

### 9.1 主题分类

输入：

```text
content_text
```

输出：

```text
topic_label
```

标签包括：

```text
prepared_food_authenticity：预制菜与食材真实性
price_value：价格与性价比
food_safety_child_meal：食品安全与儿童餐
brand_pr_crisis：品牌公关与回应翻车
luoyonghao_persona：罗永浩个人与较真形象
consumer_right_to_know：消费者知情权
catering_industry_general：餐饮行业共性讨论
mockery_meme：调侃玩梗
other_unclear：其他/不明确
```

### 9.2 情绪分类

输入：

```text
content_text
```

输出：

```text
sentiment_label
```

标签包括：

```text
positive：正面
negative：负面
neutral：中性
sarcastic：调侃/讽刺
```

### 9.3 立场分类

输入：

```text
content_text + topic_label
```

输出：

```text
stance_label
```

标签包括：

```text
question_xibei：质疑西贝
support_xibei：支持西贝
support_luoyonghao：支持罗永浩
question_luoyonghao：质疑罗永浩
neutral_discussion：中立讨论
unclear_meme：不明确/玩梗
```

立场分类建议把 `topic_label` 作为辅助输入，因为同一句话在不同主题语境下可能有不同指向。

## 10. 推荐流程

完整流程建议如下：

```text
1. 从 contents 中读取全量评论
2. 使用规则方法生成 topic / sentiment / stance 初始标签
3. 按抽样策略选择一批评论交给 DeepSeek 校正
4. 人工抽检部分高赞、低置信度和少数类样本
5. 形成训练集、验证集和测试集
6. 微调 MacBERT 主题分类模型
7. 微调 MacBERT 情绪分类模型
8. 微调 MacBERT 立场分类模型
9. 使用 MacBERT 对全量评论预测
10. 对低置信度、高赞和冲突样本再次复核
11. 生成最终文本标签表
12. 将文本标签与 relations 合并，计算信息茧房相关指标
```

第一版可以先训练三个独立模型：

```text
topic_model
sentiment_model
stance_model
```

后续如果时间充足，再考虑共享 MacBERT 编码层的多任务模型。

## 11. 训练数据建议

当前已有的模块 3、模块 4 结果可以作为起点：

```text
data/analysis/module3_topic/content_topics.csv
data/analysis/module4_sentiment_stance/content_sentiment_stance.csv
```

但现有标注规模较小，约 455 条，且类别不均衡。

因此，它们适合用于：

```text
1. 验证训练流程是否跑通
2. 构建第一版弱标注样本
3. 作为后续扩充标注的种子数据
```

不建议仅依赖这 455 条数据直接训练最终模型。

建议后续扩充到：

```text
主题分类：至少 2,000 到 5,000 条
情绪分类：至少 2,000 条
立场分类：至少 2,000 条
少数类标签每类尽量不少于 100 到 200 条
```

如果少数类样本不足，需要采用：

```text
过采样
类别权重
定向抽样
人工补标
合并过稀标签
```

## 12. 抽样策略

DeepSeek 校正和人工抽检不应完全随机抽样。

建议混合抽样：

```text
1. 高赞样本：覆盖传播影响较大的评论
2. 视频分层样本：保证每个 source_id 都有样本
3. 规则标签分层样本：保证每个 topic / sentiment / stance 候选都有样本
4. 低置信度样本：重点校正规则难判断文本
5. 多候选标签样本：处理主题混合和立场冲突
6. 少数类候选样本：补足 support_xibei、question_luoyonghao 等稀缺类别
7. 回复样本：保证楼中楼互动语境不被忽略
```

这样得到的训练集更适合支撑后续信息茧房分析。

## 13. 最终输出表

建议最终生成一张统一文本标注表：

```text
data/modeling/final_content_labels.csv
```

建议字段包括：

```text
content_id
source_id
content_type
content_text
created_at
like_count
topic_label
topic_confidence
sentiment_label
sentiment_confidence
stance_label
stance_confidence
label_method
need_review
review_reason
```

其中 `label_method` 可以记录：

```text
rule
rule+deepseek
macbert
macbert+deepseek_review
manual_review
```

这样可以保证每条标签都有来源，便于后续解释和复查。

## 14. 信息茧房指标设计

文本标签生成后，可以进入信息茧房倾向分析。

建议指标包括：

### 14.1 主题集中度

用于判断评论区是否围绕少数议题高度聚集。

可计算：

```text
最大主题占比
Top3 主题占比
主题熵
```

### 14.2 情绪一致性

用于判断评论区是否存在明显情绪共振。

可计算：

```text
主导情绪占比
negative + sarcastic 占比
情绪熵
```

### 14.3 立场一致性

用于判断评论区是否存在明显立场倾斜。

可计算：

```text
主导立场占比
质疑西贝占比
支持西贝占比
立场熵
```

### 14.4 高赞评论放大效应

用于判断被顶上去的评论是否更集中于某一主题、情绪或立场。

可比较：

```text
全量评论标签分布
高赞评论标签分布
高赞评论与普通评论的分布差异
```

### 14.5 互动同质性

用于判断用户回复是否主要发生在相似立场之间。

可计算：

```text
同立场回复比例
跨立场回复比例
同主题回复比例
同情绪回复比例
不同立场之间的回复矩阵
```

## 15. 结论表达方式

最终结论不建议简单写成：

```text
存在信息茧房
不存在信息茧房
```

更稳妥的表达是分层判断：

```text
从主题层面看，评论区是否呈现议题集中；
从情绪层面看，评论区是否呈现负面或讽刺情绪共振；
从立场层面看，评论区是否呈现明显立场倾斜；
从互动层面看，回复关系是否更倾向于同立场互动。
```

如果多个层面的证据同时成立，可以表述为：

```text
该评论区呈现一定的事件型信息茧房或局部同温层倾向。
```

如果只有部分证据成立，应表述为：

```text
该评论区在议题和情绪上存在集中化现象，但互动结构中仍保留跨立场讨论，因此不宜判断为封闭型信息茧房。
```

## 16. 第一阶段落地任务

第一阶段不追求一步到位，而是先跑通闭环。

建议任务如下：

```text
1. 生成全量规则初标结果
2. 设计 DeepSeek 分层抽样策略
3. 扩充 2,000 条左右弱标注样本
4. 人工抽检 100 到 200 条关键样本
5. 训练第一版 MacBERT 分类模型
6. 对全量评论生成 topic / sentiment / stance 标签
7. 重新计算主题、情绪、立场和互动同质性指标
8. 根据结果修订信息茧房分析结论
```

这一阶段完成后，项目就可以从“规则和 LLM 原型分析”升级为：

```text
有明确研究问题、有文本标注方法、有本地分类模型、有互动网络指标支撑的评论区舆论结构分析。
```
